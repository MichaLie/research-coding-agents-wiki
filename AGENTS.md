# AGENTS.md — AI Coding & Data Agents for Researchers

This mirrors **[CLAUDE.md](CLAUDE.md)**, which is the full canonical operating guide — read it before editing. This file restates the non-negotiables so any agent (including those that read only `AGENTS.md`) gets the critical rules.

This is an **ELIXIR-CZ**, **privacy-first** resource. Accuracy about data protection is the entire point: a wrong cell can lead a researcher to mishandle sensitive (health/genetic/clinical) data. **Treat every edit as safety-critical.**

## The four non-negotiables

1. **Currency is mandatory — never trust training memory or stale/cached docs.** Products rename, get acquired, change data policy, or shut down weekly. Confirm **every** fact against the vendor's **own current** pages (docs, pricing, privacy/trust/DPA) via live web search at edit time. Stamp `verified` with today's date; keep `sources`. (Real lesson: training-era priors were wrong — a listed tool had shut down, another was being retired/renamed; only live checks caught it. Assume your priors are stale.)

2. **Data-policy claims are safety-critical — be extremely cautious.** `data_handling`, `tier_gate`, `suitability` are the highest-stakes fields. Read the vendor's **own** privacy/DPA/trust page (not blogs, not summaries, not memory). Capture **tier nuance** — what the default/free plan does vs what the safe option requires. **Default to the more conservative classification when unsure; never guess optimistically.**

3. **Maintenance = discover AND re-check everything.** Not just add new tools — **re-verify every existing tool each pass** for changes (data policy, pricing, renames, discontinuations). Drop the dead, update the changed, downgrade claims that no longer hold.

4. **Curated, not exhaustive.** Useful > complete. Do **not** re-add deliberately-omitted things: code-review/PR bots, app-builders/"vibe-coding", autonomous *science* agents (they belong in the Autonomous STEM Agents wiki), knowledge/lit tools, abandoned projects (maintenance-health gate — popularity doesn't save a dead repo), thin forks, and obscure/enterprise-only niche tools. Keep a niche tool only if it *uniquely* serves a privacy need (e.g. the only self-host option for sensitive data).

## Special-category rule (strict)
Special-category (health/genetic/clinical, GDPR Art. 9) suitability is ✓/⚠ **only** for tools running on **infrastructure you control** (local/self-host + commercial on-prem/air-gapped). **Pure cloud is ✗ — even with ZDR/HIPAA-BAA** — because data still leaves your environment and most DUAs/IRBs forbid it. On-prem exceptions go in the note, not as a ⚠.

## Mechanics
- `tools.json` is the only hand-edited file; `docs/index.html` is generated by `python3 build_html.py tools.json docs/index.html` — never hand-edit it. Schema + full procedure: see [CLAUDE.md](CLAUDE.md) and the **`update-coding-agents-wiki`** skill.
- OSS quality gate = **maintenance health (last-commit recency), not stars**. Full re-sweep = multi-agent Workflow (segment discovery → per-tool web re-verification).
